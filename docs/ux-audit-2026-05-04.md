# UX Audit — Metaplex Agent Template

**Date:** 2026-05-04
**Branch reviewed:** `main` at `641ec29` (post-SIWS auth, post-funding-tooling)
**Reviewer:** UX / product designer pass

---

## Original prompt

> I want you to review this project as a UX expert and product designer. I want you to come up with a concrete and comprehensive list of improvements to make to this template to reduce any and all friction for both developers and non-technical users to launch their own agent. Classify things based on end user it benefits, level of effort, and impact. We want this template to be as easy to use as possible and we want it to be extremely easy for people to build and launch their own agents on Metaplex.

This document is the response, intended for sharing with stakeholders for further review.

---

## Executive summary

PR #4 closed the auth-shape gap (SIWS replaces the shared `WEB_CHANNEL_TOKEN`; three-tier wallet allowlist; no token leakage). What remains is mostly **time-to-first-chat for new operators** and **time-to-first-tool for new developers** — both still measured in tens of minutes and dozens of steps.

The single biggest remaining gap is that **launching a custom agent still requires editing TypeScript** (system prompt, tool selection, worker cadence). For the "non-technical operator" persona, that's the wall.

The fixes below would compress first-run from ~30–60 min to ~5 min, and make persona/tool customization no-code for operators.

---

## Personas (audience legend)

- **OP** — Non-technical agent operator (founder, brand, community manager) launching their own agent.
- **DEV** — Developer extending the template (custom tools, prompts, integrations).
- **DEPL** — Person deploying / operating the agent in production.
- **EU** — End user chatting with a deployed agent.

## Scoring legend

- **Effort:** S = ≤1 day, M = ~1 week, L = 2–4 weeks
- **Impact:** ★ low, ★★ medium, ★★★ high

## P0 — Highest-leverage wins

| # | Recommendation | Audience | Effort | Impact |
|---|---|---|---|---|
| 1 | **`npx create-metaplex-agent` scaffolder.** Interactive CLI: pick mode, name, persona preset, LLM provider; auto-generates `AGENT_KEYPAIR` (don't make users learn `solana-keygen`); seeds `wallets.allowlist.json` with the operator's own wallet; emits a hosted-UI deep link as the final output. Eliminates ~10 manual steps. | OP, DEV | M | ★★★ |
| 2 | **One-click "Deploy to Railway" button in the README.** Pre-fills `AGENT_KEYPAIR`, `AGENT_MODE`, `WALLET_ALLOWLIST`, RPC, LLM key. End-state: paste keypair, press button, get a `wss://` URL. The Railway path already works — just package it. | OP, DEPL | S | ★★★ |
| 3 | **Persist registration to env automatically on Railway/Fly/Render.** Today the operator must hand-copy `AGENT_ASSET_ADDRESS` from logs to env after first run, or lose identity on every redeploy. Either auto-attach a Railway volume in `railway.json`, or on first registration log a copy-pasteable env diff plus a friendly red banner; ideally call the Railway API to set the var. Silent footgun otherwise. | DEPL | S–M | ★★★ |
| 4 | **`config.yaml` for persona & behavior** (optional, layered over `.env`). Operators set agent name, system prompt, enabled tools, worker cadence, dry-run, per-tool caps in one human-readable file. Today persona lives in `prompts.ts` and gets clobbered by `pnpm bootstrap`. | OP | M | ★★★ |
| 5 | **Compress `.env.example` to ~30 lines for the happy path.** Current file is 200+ lines including all the SIWS / rate-limit knobs. Move tunables into `.env.advanced.example` or docs. Required-only for v1 = mode, keypair, RPC, LLM key, optionally `WALLET_ALLOWLIST`. | OP, DEV | S | ★★★ |
| 6 | **Hosted live demo + 60-second screencast in the README.** "Click here, chat with a Metaplex agent right now" + a GIF of the wallet → SIWS popup → chat → tx-approval loop. | 7 | **`pnpm doctor` diagnostic.** Validates `.env`, checks RPC reachability, verifies LLM API key works, prints agent's wallet addresses + balances, runs the existing `smoke-siws.ts` against the local server, and warns on Railway about ephemeral fs. Replaces the painful "why won't it start?" loop. | OP, DEV | S | ★★★ |
| 8 | **Single-command dev (`pnpm dev` runs server + UI together).** UI lives in a sibling repo. Add `pnpm dev:full` that `git clone`s the chat-template on first run, runs both with `concurrently`. With SIWS, no token-mirror step is required anymore — only the WSS URL. | OP, DEV | S | ★★ |

| 14 | **Built-in test harness.** Today only `@metaplex-agent/shared` has tests (38 from the SIWS work). Extend Vitest/`node:test` setup into `core` and `server` with example tool tests and a mock `submitOrSend`. Forks inherit testing-as-default. | DEV | M | ★★ |
| 15 | **Health/status HTTP endpoint.** `GET /health` returning `{ mode, registered, owner, lastTickAt, paused, errorStreak, dryRun, authMode }` so platforms (Railway/Fly health checks, uptime monitors) can probe without a WS+SIWS connection. | DEPL | S | ★★ |

## What's already strong (preserve in any redesign)

- **SIWS auth + tiered allowlist** is the structural improvement; the rest of this list builds on it.
- **Transaction-approval modal** is genuinely well-built (decoded preview, focus trap, Escape, success/error states, copy + explorer link). Don't regress it.
- **`pnpm bootstrap`** mode-pruner — keep, extend with `--persona <preset>`.
- **`AUTONOMOUS_DRY_RUN=true`** default and the byte-for-byte canonical SIWS verification are exemplary safety design.
- **`scripts/smoke-siws.ts`** — a real diagnostic primitive that #7 (`pnpm doctor`) can chain into.
- **The `[auth-failure]` structured-log pattern** is the right shape; #16 just generalizes it.
- **The Zod-validated config** catches most footguns at startup.
- **The autonomous worker loop's owner-gating + error-streak auto-pause** is solid defensive design.

## Out of scope for this audit (intentional omissions)

- **LLM provider quality / model selection** — this is downstream of the template's job.
- **Solana program-level changes** to the Agent Registry or Genesis — separate teams.
- **Marketing copy** for agents.metaplex.com — referenced where it touches discoverability (#31, #32) but not detailed.
- **Cost / pricing model** for hosted agents — out of scope.
- **Detailed per-deploy-target guides** beyond Railway (Fly, Render, Koyeb, k8s) — once the Railway button works, the others follow the same pattern; covered already in `docs/DEPLOYMENT.md`.
