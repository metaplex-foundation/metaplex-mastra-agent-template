## Original prompt

> I want you to review this project as a UX expert and product designer. I want you to come up with a concrete and comprehensive list of improvements to make to this template to reduce any and all friction for both developers and non-technical users to launch their own agent. Classify things based on end user it benefits, level of effort, and impact. We want this template to be as easy to use as possible and we want it to be extremely easy for people to build and launch their own agents on Metaplex.

This document is the response, intended for sharing with stakeholders for further review.

---

## Executive summary

PR #4 closed the auth-shape gap (SIWS replaces the shared `WEB_CHANNEL_TOKEN`; three-tier wallet allowlist; no token leakage). **Sprint 1 (PR #5) followed and delivered the time-to-first-chat collapse**: `pnpm setup` interactive scaffolder, `pnpm doctor` pre-flight diagnostic, `pnpm dev:full` single-command dev (clones the chat-template sibling on first run), Railway deploy button, slim `.env.example` (-83%), tiered README (-78%) with a new `docs/ARCHITECTURE.md`. The chat-template repo's PR #2 follow-up cleaned up the legacy `WEB_CHANNEL_TOKEN` profile field from its README to match the SIWS-wired UI.

The single biggest remaining gap is that **launching a *custom* agent still requires editing TypeScript** (system prompt, tool selection, worker cadence). For the "non-technical operator" persona, that's the wall — and the focus of Sprint 2.

The remaining (un-checked) items below would make persona/tool customization no-code for operators and turn the template into a launchable platform rather than a starter kit.

## Personas (audience legend)

- **OP** — Non-technical agent operator (founder, brand, community manager) launching their own agent.
- **DEV** — Developer extending the template (custom tools, prompts, integrations).
- **DEPL** — Person deploying / operating the agent in production.
- **EU** — End user chatting with a deployed agent.

## Scoring legend

- **Effort:** S = ≤1 day, M = ~1 week, L = 2–4 weeks
- **Impact:** ★ low, ★★ medium, ★★★ high

## What's already strong (preserve in any redesign)

- **SIWS auth + tiered allowlist** is the structural improvement; the rest of this list builds on it.
- **Transaction-approval modal** is genuinely well-built (decoded preview, focus trap, Escape, success/error states, copy + explorer link). Don't regress it.
- **`pnpm setup` + `pnpm doctor` + `pnpm dev:full`** (Sprint 1, PR #5) are now part of the canonical onboarding flow; preserve them in any future scaffolder rewrite.
- **`pnpm bootstrap`** mode-pruner — keep, extend with `--persona <preset>`.
- **Tiered docs** (`README.md` quickstart + `docs/ARCHITECTURE.md` deep-dive) — preserve the split when adding new content.
- **`AUTONOMOUS_DRY_RUN=true`** default and the byte-for-byte canonical SIWS verification are exemplary safety design.
- **`scripts/smoke-siws.ts`** — diagnostic primitive that `pnpm doctor` already chains into; useful for CI too.
- **The `[auth-failure]` structured-log pattern** is the right shape; #16 just generalizes it.
- **The Zod-validated config** catches most footguns at startup.
- **The autonomous worker loop's owner-gating + error-streak auto-pause** is solid defensive design.
